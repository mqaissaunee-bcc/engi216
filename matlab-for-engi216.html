Motivation · ~10 minWhy this course leans on MATLAB

Mechanisms produce equations that are easy to write down and miserable to solve by hand. MATLAB exists to do the miserable part instantly and reliably.

The closed-form wall

Chapter 2 of your text draws a hard line. A closed-form solution lets you compute the unknowns directly — no iteration. But as the lecture states, position equations are typically nonlinear (full of sines and cosines), and only special cases like the four-bar or crank-slider have tidy closed forms. For almost everything else, the slide is blunt: “most mechanisms do not have a closed-form solution — we will use Newton's method.”

Newton's method means iterating: guess, evaluate, correct, repeat. Doing that by hand for a four-bar across 360° of crank rotation — 73 angles × several iterations each — is hundreds of evaluations. MATLAB does it in milliseconds.

Real world · Why the bridge engineer reaches for MATLAB Example 8.1 in your text designs a four-bar mechanism to lift an urban drawbridge, replacing a heavy 2:1 gear pair. The designer solves a 3×3 linear system for the link-length ratios, then plots the output angle against the input angle to check whether the mechanism actually tracks the desired motion. Then they tweak the precision points and re-run. Without MATLAB, each redesign is an afternoon of arithmetic; with it, it's a keystroke. That iterate-and-check loop is the real workflow of a machine designer.

Part 1 · ~25 minThe absolute basics

Variables, arithmetic, the semicolon, and reading errors. Master these and the rest is detail.

Assigning variables and the all-important semicolon

You assign with =. If a line ends without a semicolon, MATLAB prints the result. End it with ; and it stays silent. In a 73-step loop, forgetting the semicolon floods your screen with 73 printouts — the single most common beginner annoyance.

Command Window
% Link lengths of a four-bar (feet) -- semicolons keep it quiet
r1 = 10;        % ground
r2 = 6.866;     % crank
r3 = 17.278;    % coupler
r4 = 16.886;    % rocker

theta2 = 45          % no semicolon -> MATLAB echoes: theta2 = 45

Degrees vs. radians — the error that wrecks mechanism code

MATLAB's sin, cos, tan expect radians. Your textbook states angles in degrees. Mixing them produces answers that are silently, catastrophically wrong — no error message, just nonsense. Convert deliberately:

Command Window
% WRONG: feeding degrees straight into cos
cos(45)              % = 0.5253  (cosine of 45 RADIANS -- garbage)

% RIGHT: convert first
cosd(45)             % = 0.7071  (the 'd' versions take DEGREES)
cos(45 * pi/180)    % = 0.7071  (manual conversion, same answer)
Pitfall · The #1 silent bug in this course A four-bar that “won't close” or gives impossible angles is almost always a degree/radian mix-up. Decide once: work in radians internally, convert to degrees only when printing/plotting — OR use the sind/cosd/atan2d family everywhere. Don't mix the two styles in one file.

Part 2 · ~30 minVectors & matrices — the heart of it

Every mechanism in this course is a loop of position vectors. MATLAB was built for exactly this. Get comfortable here and Chapters 2–8 stop being scary.

Building them: rows, columns, and the magic of :

Command Window
% Spaces (or commas) -> across a row. Semicolons -> new row.
row  = [3 4 5];        % 1x3 row vector
col  = [3; 4; 5];      % 3x1 column vector
A    = [2 1; -1 3];     % 2x2 matrix (rows split by ;)

% The colon makes ranges -- this is how you sweep a crank angle:
theta2 = 0:5:360;       % 0,5,10,...,360  (73 values) in ONE line
n = numel(theta2);     % = 73, how many elements
Analogy · A matrix is a spreadsheet, a vector is one of its columns If you can picture a spreadsheet, you already understand MATLAB matrices. A(2,3) is “row 2, column 3” — exactly like clicking a cell. A vector is just a spreadsheet with one row or one column. The colon range 0:5:360 is like dragging the fill-handle down to auto-fill a sequence.

Indexing: reaching into a vector or matrix

Command Window
v = [10 20 30 40];
v(2)       % = 20    (MATLAB counts from 1, NOT 0!)
v(end)     % = 40    'end' = last element
v(2:3)     % = [20 30]   a slice

A = [2 1; -1 3];
A(1,2)     % = 1     row 1, column 2
A(:,1)     % = [2; -1]   the whole 1st column (':' means "all")
Pitfall · MATLAB indexes from 1 Coming from Python, C, or Java? In those, the first element is v[0]. In MATLAB it's v(1), with parentheses, counting from one. Asking for v(0) is an error. This trips up nearly everyone for the first week.

The crucial distinction: * vs .*

This is the concept that separates people who fight MATLAB from people who fly in it.

  • * is matrix multiplication — the linear-algebra operation, with strict dimension rules.
  • .* (with a dot) is element-by-element — multiply each pair of matching entries.

The same holds for ./ and .^. When you want to compute, say, r2*cosd(theta2) for all 73 crank angles at once, you need element-wise so each angle pairs with itself.

Command Window
theta2 = 0:90:270;          % [0 90 180 270]
r2 = 6;

% Element-wise: one x-coordinate per angle (a 1x4 result)
x = r2 .* cosd(theta2)     % = [6  0  -6  0]   -- exactly what we want

% Try plain * here and MATLAB errors: inner dimensions disagree.
Analogy · .* is an assembly line, * is a committee Element-wise .* is an assembly line: item 1 meets item 1, item 2 meets item 2, independently, in parallel. Matrix * is a committee meeting where every row negotiates with every column and the results get summed. For sweeping a value across many angles, you want the assembly line.

Transpose, magnitude, and dot product

Command Window
v = [3 4];
v'              % transpose: turns the row into a column
norm(v)         % = 5    magnitude (Pythagoras) -- a vector's LENGTH
dot([1 0],[0 1])  % = 0    perpendicular vectors -> zero dot product
Real world · The vector loop, in MATLAB terms Your text's vector-loop method says the position vectors around a closed mechanism sum to zero: r₂ + r₃ − r₄ − r₁ = 0. In MATLAB each vector is just [r*cosd(th), r*sind(th)]. The “loop closes” when the sum's norm is essentially zero. That single idea — measure how badly the loop fails to close, then fix it — is the entire engine of numerical position analysis in Part 5.

Part 3 · ~25 minSolving a linear system: A\b

This single operator — the backslash — powers closed-form position analysis and the whole of Freudenstein synthesis in Chapter 8. Learn it well; you'll use it on every synthesis problem.

The problem MATLAB is solving

A system of linear equations like

the math
  2x +  y = 8
   x + 3y = 10

is written in matrix form as A·x = b, where A holds the coefficients, x the unknowns, and b the right-hand side. To solve it in MATLAB:

Command Window
A = [2 1; 1 3];     % coefficient matrix
b = [8; 10];        % right-hand side (a COLUMN)
x = A\b              % the backslash solves A*x = b  ->  x = [2.8; 2.4]

That's it. The backslash (officially mldivide) runs Gaussian elimination under the hood and hands you the answer. Per the MathWorks documentation, it factorizes A and is far more efficient and accurate than computing an inverse.

Pitfall · Use A\b, never inv(A)*b It's tempting to write x = inv(A)*b — it looks like the textbook's x = A⁻¹b. Don't. Forming the inverse explicitly is slower and less numerically accurate. The MathWorks docs are explicit that the inverse approach is “the most obvious (and worst)” option. Backslash is the professional habit.
Analogy · Backslash is GPS routing, inverse is mapping every road first To get from A to B, you don't first compute the optimal route between every pair of intersections in the city and then look yours up — that's what inv(A) does. Backslash is the GPS that solves only your trip, directly and fast. Same destination, a fraction of the work.

Chapter 8 in action: Freudenstein point-matching

Here's the actual payoff. Example 8.1 designs a four-bar to mimic a 2:1 gear pair. After converting three precision points, the text forms this exact 3×3 system (its Equation 8.11) for the link ratios R₁, R₂, R₃:

% Stanisic Example 8.1 -- solve for the three link-length ratios
A = [-0.423  0.462  1;
      0.342  0.087  1;
      0.906 -0.301  1];

Part 4 · ~25 minScripts, loops, and functions

Homework isn't one line — it's a reusable solution. Here's how to package your work so you can re-run it, sweep it across angles, and hand it in.

Scripts: your saved recipe

A script is a .m file full of commands that run top-to-bottom when you press Run. Start every script with clear; clc; to wipe old variables and clear the screen — otherwise stale values from a previous run can poison your answer.

analyze_crank.m
clear; clc;                 % fresh start -- do this at the top of every script

r2 = 6;
theta2 = 30;
x = r2 * cosd(theta2);
y = r2 * sind(theta2);

fprintf('Crank tip at (%.3f, %.3f)\n', x, y);  % formatted print

for loops: doing it 73 times

To analyze a mechanism through a full crank rotation, you loop. Crucially, preallocate the results array first — it's faster and signals your intent.

sweep_crank.m
angles = 0:5:360;        % 73 crank positions
n = numel(angles);
xtip = zeros(1, n);      % PREALLOCATE: a row of 73 zeros to fill in

for i = 1:n
    th = angles(i);
    xtip(i) = 6 * cosd(th);   % store result in slot i
end

% Bonus: this whole loop is ONE vectorized line --
xtip = 6 * cosd(angles);    % same result, no loop (element-wise!)
Analogy · Preallocating is reserving the parking lot Growing an array inside a loop (x(i)=... when x didn't exist) forces MATLAB to keep finding bigger parking lots and moving every car each time a new one arrives. zeros(1,n) reserves all 73 spaces up front so each result just pulls into its assigned spot. On a 73-step sweep you won't notice; on a 100,000-step simulation it's the difference between seconds and minutes.

Functions: the reusable black box

A function takes inputs, returns outputs, and keeps its own private variables. This is how you write a position-analysis routine once and call it for any four-bar. The function line names the file.

fourbar_pos.m
function [theta3, theta4] = fourbar_pos(r1, r2, r3, r4, theta2)
% Returns coupler & rocker angles (deg) for a four-bar in open config.
    th2 = deg2rad(theta2);
    % Position of the crank pin:
    Ax = r2*cos(th2);  Ay = r2*sin(th2);
    % Distance from that pin to the fixed rocker pivot at (r1,0):
    d = hypot(Ax - r1, Ay);
    % Law of cosines for the transmission geometry:
    beta  = acos((r3^2 + d^2 - r4^2) / (2*r3*d));
    psi   = atan2(Ay, Ax - r1);
    theta3 = rad2deg(psi + beta);
    % Rocker angle from its own pivot:
    Bx = Ax + r3*cosd(theta3);  By = Ay + r3*sind(theta3);
    theta4 = atan2d(By, Bx - r1);
end
Command Window — calling it
[t3, t4] = fourbar_pos(10, 6.866, 17.278, 16.886, 45)
% Now reusable for ANY crank angle, or inside a sweep loop.
Anonymous functions — one-liners you'll need for solvers For quick math you don't want in a separate file, use an anonymous function with @: f = @(x) x.^2 - 2; then f(3) gives 7. The @(x) means “a function of x.” You'll hand exactly this kind of one-liner to fzero and fsolve in the next part.
Real world · Functions are how real engineering code is organized No working mechanism analysis lives as one giant script. A vehicle suspension model might call fourbar_pos thousands of times across a road-input simulation. Writing it once as a tested function — rather than copy-pasting the geometry — is the difference between code you trust and code you fear. Your ENGI 216 homework is where that habit starts.

Part 5 · ~35 minNonlinear equations: Newton's method & solvers

Chapter 2.4 is explicit: most mechanisms have no closed form, so we solve their position equations numerically with Newton's method. Here you'll see it by hand and with MATLAB's built-in solvers — on a real four-bar loop.

The idea, in one sentence

Newton's method finds where a function equals zero by repeatedly following the tangent line down to the axis: guess → measure the error → use the slope to correct → repeat until the error is tiny. Your lecture notes that it “converts the original system of nonlinear equations into a system of approximate linear equations” — that linearization is the tangent step.

the update rule
  x_new = x_old  -  f(x_old) / f'(x_old)      (one unknown)
  x_new = x_old  -  J\f(x_old)                (several unknowns; J = Jacobian)
Analogy · Rolling a marble down to the valley floor Imagine the function as a curved ramp and you want the spot where it touches the floor (f = 0). From wherever you stand, you look at the slope under your feet and step in the downhill direction by just the right amount to hit the floor if the ramp were straight. It isn't straight, so you overshoot a little — then repeat from the new spot. Each step the ramp looks straighter, so you converge fast. Near the answer, Newton's method roughly doubles the number of correct digits every step.

By hand: solving f(x) = x² − 2 = 0

Before mechanisms, watch the mechanics on a problem whose answer you know: √2 ≈ 1.41421. Here f(x)=x²−2 and f'(x)=2x.

newton_by_hand.m
f  = @(x) x^2 - 2;     % the function
df = @(x) 2*x;          % its derivative

x = 2;                   % initial guess
for k = 1:6
    x = x - f(x)/df(x);  % the Newton step
    fprintf('iter %d:  x = %.8f\n', k, x);
end
% Converges to 1.41421356 in ~4 steps -- watch it in the widget below.

Stepping up: Newton for a system (the Jacobian)

One equation used the slope f'(x). A four-bar's position has two coupled equations in two unknowns (the coupler angle θ₃ and rocker angle θ₄), so the single slope becomes a 2×2 matrix of partial derivatives called the Jacobian, J. This is the part Chapter 2.4 leans on and the part most likely to show up on an exam, so let's derive it explicitly.

Write the loop closure as two functions that must both equal zero (x- and y-components):

f₁=r₂cosθ₂ + r₃cosθ₃ − r₄cosθ₄ − r₁ = 0
f₂=r₂sinθ₂ + r₃sinθ₃ − r₄sinθ₄ = 0

The unknowns are θ₃ and θ₄. Take the partial derivative of each equation with respect to each unknown — that's four derivatives, arranged in a 2×2 grid. Remember d/dθ(cosθ) = −sinθ and d/dθ(sinθ) = cosθ:

J= ∂f₁/∂θ₃∂f₁/∂θ₄ ∂f₂/∂θ₃∂f₂/∂θ₄ = −r₃sinθ₃+r₄sinθ₄ +r₃cosθ₃−r₄cosθ₄
Analogy · the Jacobian is a 2-D slope For one variable, the slope tells you “if I nudge x, how much does f move?” The Jacobian answers the same question with two inputs and two outputs at once: “if I nudge θ₃ and θ₄, how do both equations respond?” Each entry is one of those four cross-sensitivities. It's a slope that's been promoted to a matrix.

The multi-variable Newton step is the direct analogue of the scalar one — division by the slope becomes a backslash solve against J (here's where Part 3 pays off):

newton_system.m
% Four-bar at theta2 = 45 deg. Unknowns th3, th4 (radians).
r1=10; r2=6.866; r3=17.278; r4=16.886; th2=deg2rad(45);

t = [deg2rad(20); deg2rad(80)];   % initial guess [th3; th4]

for k = 1:6
    th3 = t(1); th4 = t(2);
    % the two loop-closure residuals:
    f = [ r2*cos(th2) + r3*cos(th3) - r4*cos(th4) - r1;
          r2*sin(th2) + r3*sin(th3) - r4*sin(th4) ];
    % the Jacobian, exactly as derived above:
    J = [ -r3*sin(th3),  r4*sin(th4);
           r3*cos(th3), -r4*cos(th4) ];
    t = t - J\f;            % the Newton step:  J \ f  (NOT f/J)
    fprintf('iter %d:  th3 = %.4f deg,  th4 = %.4f deg,  |f| = %.2e\n', ...
            k, rad2deg(t(1)), rad2deg(t(2)), norm(f));
end
One iteration, by hand — so you trust the loop Start at θ₃=20°, θ₄=80°. Plug into f: the loop doesn't close, so f ≈ [8.16; −5.86] (far from zero). Build J from the same angles, then solve the 2×2 system J·Δ = −f with backslash to get the correction Δ = −J\f. Add it: t = t + Δ. The residual |f| shrinks dramatically each pass. Repeat a few times and it's machine-zero (the converged angles are θ₃≈31.6°, θ₄≈55.5°). That is the entire algorithm — everything fsolve does, you just did by hand.
Pitfall · it's J\f, never f/J or f./J The Newton step solves the linear system J·Δ = −f for the correction vector Δ. That's a left division: J\f. Writing f/J (right division) or f./J (element-wise) silently gives the wrong vector. When in doubt, recall: the unknown is sandwiched as J·Δ, so Δ comes out by backslashing J from the left.

The built-in shortcuts: fzero and fsolve

Now that you've built Newton by hand, here's the good news: fsolve does all of that — including estimating the Jacobian for you — from just the residual function and a starting guess. You hand-code Newton to understand it; you call fsolve to use it. MATLAB ships robust solvers for both the scalar and system cases.

Which solver when
SolverUse forNeeds toolbox?
fzeroone equation in one unknown (e.g. one scalar loop equation)No — built into base MATLAB & Octave
fsolvea system of nonlinear equations (e.g. two coupled loop equations)Optimization Toolbox (campus licenses usually include it)
solver_demo.m
% Single equation: same sqrt(2) problem, zero hand-coding
root = fzero(@(x) x^2 - 2, 2)     % = 1.4142, guess = 2

% System: a four-bar's two scalar loop equations (x- and y-components).
% Unknowns are theta3, theta4 (radians). Loop must close -> both = 0.
r1=10; r2=6.866; r3=17.278; r4=16.886; th2=deg2rad(45);
loop = @(t) [ r2*cos(th2) + r3*cos(t(1)) - r4*cos(t(2)) - r1;
               r2*sin(th2) + r3*sin(t(1)) - r4*sin(t(2)) ];
sol = fsolve(loop, [0.5; 1.5]);     % starting guess for [th3; th4]
disp(rad2deg(sol));                  % the coupler & rocker angles, in degrees
Pitfall · The initial guess matters A four-bar has two valid assembly configurations (“open” and “crossed”). The solver returns whichever one your starting guess is closest to. If you get the wrong branch, nudge the guess toward the configuration you want. And when sweeping across angles, feed each step's answer as the next step's guess — the mechanism moves only a little between angles, so the previous solution is an excellent starting point and keeps you on one consistent branch.
Real world · Why “continuation” matters in engine simulation When engineers simulate a piston engine through a full cycle, they don't re-guess the geometry from scratch at each crank angle — they march forward, each solution seeding the next. This “continuation” trick is exactly the warm-start habit above, and it's why a well-written sweep never jumps to the wrong configuration mid-rotation. You'll use it in the capstone.

Part 7 · ~30 minVelocity & acceleration: the same matrix, twice

Once position is solved, Chapter 4 wants velocities and accelerations — and here's the payoff you've been building toward: they come from solving a linear system with the same Jacobian you just derived. It's A\b all over again.

Where the linear system comes from

Chapter 4.1 says it directly: “taking derivatives and rearranging, you go from the position equations to linear systems describing velocity and acceleration.” Differentiate the two loop-closure equations with respect to time. The crank rate θ̇₂ is known (it's the input); the unknowns θ̇₃ and θ̇₄ appear linearly, multiplied by exactly the Jacobian entries from Part 5:

J θ̇₃θ̇₄ = − −r₂sinθ₂+r₂cosθ₂ θ̇₂

That's a 2×2 linear system J·{velocities} = {b}. The matrix on the left is the identical J from position analysis. So once you have J, velocity analysis is one backslash.

The big idea connecting Parts 3, 5, and 7 Position (nonlinear) needed Newton, which built J. Velocity and acceleration (linear once position is known) reuse that same J and just change the right-hand side. One matrix, three jobs. If you can form J, you can do all of Chapters 2 and 4.

Two ways to read the same solve

The textbook frames this two ways, and they're the same equation with the input rate divided out. This is the heart of Chapter 4.

Time-based rates vs. kinematic coefficients — one solve, two interpretations
 Time-based (Ch. 4.1)Kinematic coefficients (Ch. 4.2)
You solveJ\b with b containing θ̇₂J\b0 with b0 = the b above but with θ̇₂ set to 1
You getactual rates θ̇₃, θ̇₄ (rad/s)coefficients h₃=dθ₃/dθ₂, h₄=dθ₄/dθ₂
Relationshipθ̇₃ = h₃ · θ̇₂ — the coefficient is the rate per unit input rate
Why botherdirect, but mixes in the input speedpure geometry — independent of how fast the crank turns
Analogy · speed vs. gear ratio Time-based rates are like saying “the output shaft turns at 50 rpm.” Kinematic coefficients are like saying “the output turns 0.5 revolutions per input revolution” — a ratio that holds no matter how fast you crank. Dividing the velocity equation by θ̇₂ converts speed into ratio, which is exactly why Chapter 4.2 says coefficients “reveal the purely geometric nature” that raw time-derivatives hide.

In MATLAB — both, from one Jacobian

velocity_analysis.m
% Assumes you've already solved position for th3, th4 at this th2.
r2=6.866; r3=17.278; r4=16.886;
th2=deg2rad(45); th2dot = 10;   % crank speed, rad/s (the input)
% th3, th4 come from your position solve (Part 5):
th3 = deg2rad(22.3); th4 = deg2rad(98.7);   % example values

% SAME Jacobian as position analysis:
J = [ -r3*sin(th3),  r4*sin(th4);
       r3*cos(th3), -r4*cos(th4) ];

% --- Way 1: time-based rates (b carries the actual crank speed) ---
b  = -[ -r2*sin(th2); r2*cos(th2) ] * th2dot;
w  = J\b;                % w = [th3dot; th4dot] in rad/s

% --- Way 2: kinematic coefficients (same b, but th2dot = 1) ---
b0 = -[ -r2*sin(th2); r2*cos(th2) ];
h  = J\b0;               % h = [h3; h4], pure geometry

% Check the relationship:  w should equal h * th2dot
disp([w, h*th2dot]);    % the two columns match

Acceleration: same matrix, third right-hand side

Differentiate once more and you get another linear system — again with the same J on the left. The only new work is the right-hand side, which now includes the velocity terms you just found:

acceleration_analysis.m
% Continues from velocity_analysis.m. th2ddot = crank angular accel.
th2ddot = 0;                    % e.g. constant crank speed -> 0
th3dot = w(1); th4dot = w(2);

% RHS gathers the known (velocity-squared) terms:
rhs = [ r2*cos(th2)*th2dot^2 + r3*cos(th3)*th3dot^2 - r4*cos(th4)*th4dot^2 + r2*sin(th2)*th2ddot;
        r2*sin(th2)*th2dot^2 + r3*sin(th3)*th3dot^2 - r4*sin(th4)*th4dot^2 - r2*cos(th2)*th2ddot ];

alpha = J\rhs;     % alpha = [th3ddot; th4ddot] -- SAME J, new RHS

Set the crank angle and speed. The widget builds J, then solves it twice — once for actual rates, once (with θ̇₂=1) for kinematic coefficients — and confirms θ̇ = h·θ̇₂.

Real world · why coefficients matter for dead positions Chapter 4.3–4.4 use kinematic coefficients to find a mechanism's limit and dead positions — the crank angles where the output momentarily stops or reverses. Those are exactly where a coefficient like h₄ passes through zero. Because coefficients are pure geometry, you can scan them across a full rotation with a loop and a plot (Part 6) to find these critical positions — essential for designing, say, a clamping mechanism that must hold firm at the end of its stroke.
Try it · in real MATLAB Wrap the velocity solve in a loop over theta2 = 0:5:360 (reusing your position solver from the capstone to get th3, th4 at each step), store h4, and plot(theta2, h4). The angles where the curve crosses zero are the dead positions. You've now combined Parts 3, 5, 6, and 7 into a genuine Chapter 4 analysis.

Part 8 · ~25 minCapstone: a complete four-bar analysis

Everything assembles here. This one script — vectors, a function, a solver with warm-starting, a loop, and a plot — is a template you can adapt for most ENGI 216 position-analysis homework.

fourbar_full_analysis.m
clear; clc;

% ---- 1. Define the mechanism (the Ex. 8.1 drawbridge linkage) ----
r1 = 10; r2 = 6.866; r3 = 17.278; r4 = 16.886;   % link lengths (ft)

% ---- 2. Sweep the crank through a full rotation ----
theta2 = 0:5:360;
n = numel(theta2);
theta3 = zeros(1,n);   theta4 = zeros(1,n);   % preallocate

guess = [deg2rad(20); deg2rad(80)];   % [th3; th4] starting guess (rad)

for i = 1:n
    th2 = deg2rad(theta2(i));
    % Vector-loop closure: x- and y- components must both vanish
    loop = @(t) [ r2*cos(th2)+r3*cos(t(1))-r4*cos(t(2))-r1;
                   r2*sin(th2)+r3*sin(t(1))-r4*sin(t(2)) ];
    sol = fsolve(loop, guess, optimset('Display','off'));
    theta3(i) = rad2deg(sol(1));
    theta4(i) = rad2deg(sol(2));
    guess = sol;        % WARM START: feed this answer to the next step
end

% ---- 3. Plot the performance curve ----
plot(theta2, theta4, 'LineWidth', 2);
xlabel('Crank \theta_2 (deg)');  ylabel('Rocker \theta_4 (deg)');
title('Four-bar rocker motion over one crank revolution');
grid on

fprintf('At theta2 = 45 deg:  theta4 = %.2f deg\n', theta4(theta2==45));
Try it · your turn Run the capstone in MATLAB. Then make three changes and predict each result first: (1) change r2 to 8 — does the rocker swing more or less? (2) Remove the guess = sol; warm-start line — watch the curve develop a discontinuous jump where the solver flips branches. (3) Add a second plot of theta3 with hold on and a legend.
Real world · This template scales to a career The structure you just built — define geometry, sweep an input, solve the closure at each step, plot the behavior — is the skeleton of professional kinematic-analysis tools, from windshield-wiper linkage design to robot-arm workspace mapping. The toolbox you've assembled in this primer (vectors, A\b, fsolve, plotting) is genuinely the working set practicing mechanism designers reach for.

PracticeWorked problems — try first, then check

Three problems in the style of ENGI 216 homework. Attempt each in MATLAB before expanding the solution. They build on each other: position → synthesis → velocity.

Problem 1 · Position analysis (Ch. 2)

Find the coupler and rocker angles of a four-bar

A four-bar has ground r1 = 12, crank r2 = 4, coupler r3 = 13, rocker r4 = 8 (all cm). For a crank angle θ₂ = 60°, find θ₃ and θ₄ in the open configuration. Use the Jacobian-Newton approach from Part 5 (or fsolve).

Show solution

Set up the two loop-closure residuals and solve from a guess near the open branch:

problem1.m
r1=12; r2=4; r3=13; r4=8; th2=deg2rad(60);
loop = @(t)[ r2*cos(th2)+r3*cos(t(1))-r4*cos(t(2))-r1;
              r2*sin(th2)+r3*sin(t(1))-r4*sin(t(2)) ];
sol = fsolve(loop, [deg2rad(30); deg2rad(90)]);
disp(rad2deg(sol));

Answer: θ₃ ≈ 18.8°, θ₄ ≈ 73.3° Verify by plugging back in — both residuals should be ~0. If you got a different pair, you likely landed on the crossed branch; nudge the guess toward the open one.

Problem 2 · Synthesis (Ch. 8)

Three-point Freudenstein function generation

You are given a 3×3 precision-point system (already assembled from three {φ,ψ} pairs) for the link ratios R₁, R₂, R₃:

−0.500.621 0.170.211 0.77−0.341 R₁R₂R₃ = 0.940.880.71

Solve for the ratios, then with assumed ground length r1 = 10 find the crank length r2 = r1/R₁ and rocker r4 = r1/R₂.

Show solution

This is a pure A\b problem — exactly the Part 3 pattern:

problem2.m
A = [-0.50  0.62  1;
      0.17  0.21  1;
      0.77 -0.34  1];
b = [0.94; 0.88; 0.71];
R = A\b;
r1 = 10;
r2 = r1/R(1);
r4 = r1/R(2);
fprintf('R = [%.4f %.4f %.4f]\n', R);

Problem 3 · Velocity (Ch. 4)

Angular velocities and the rocker kinematic coefficient

Using the four-bar from Problem 1 (r2=4, r3=13, r4=8) at θ₂=60° with your solved θ₃≈18.8°, θ₄≈73.3°, and a crank speed θ̇₂ = 15 rad/s: find the rocker angular velocity θ̇₄ and the rocker kinematic coefficient h₄. Use the same Jacobian, two right-hand sides (Part 7).

Show solution

Build J once; solve with the speed for rates, and with θ̇₂=1 for coefficients:

problem3.m
r2=4; r3=13; r4=8;
th2=deg2rad(60); th3=deg2rad(18.8); th4=deg2rad(73.3);
th2dot = 15;

J = [ -r3*sin(th3),  r4*sin(th4);
       r3*cos(th3), -r4*cos(th4) ];
b0 = -[ -r2*sin(th2); r2*cos(th2) ];

h = J\b0;          % kinematic coefficients [h3; h4]
w = h * th2dot;     % actual rates [th3dot; th4dot]
fprintf('h4 = %.4f,  th4dot = %.3f rad/s\n', h(2), w(2));

Answer: h₄ ≈ 0.40, θ̇₄ ≈ 6.1 rad/s The coefficient h₄ says the rocker turns about 0.40° per degree of crank — geometry-only. Multiply by the 15 rad/s crank speed to recover the actual rate. (Your exact values depend on the precision of the θ₃,θ₄ you carry in from Problem 1.)
